What is Leanstral?

Released March 16, 2026, it's Mistral's open-source Lean 4 theorem proving agent, built on Mistral Small 4 (119B total params, MoE with 6.5B active per token, 256K context). Trained specifically to work with Lean's language server via MCP. On FLTEval (Fermat's Last Theorem formalization benchmark), Leanstral pass@2 beats Claude Sonnet 4.6 at 15% of the cost. Apache 2.0 licensed, weights on HuggingFace. Mistral hasn't published the training details yet — a tech report is forthcoming.

Adaptation Methods in 2025–2026

LoRA is still the workhorse. It remains the dominant PEFT method. Its main successors:

  • DoRA (ICML 2024 Oral, NVIDIA) — decomposes weights into magnitude + direction, applies LoRA only to direction. Consistently outperforms LoRA, especially at low ranks. Now in HuggingFace PEFT.
  • QLoRA — LoRA + 4-bit quantized base model. Fine-tune a 7B model on a single RTX 4090.

A NeurIPS 2025 paper showed LoRA and full fine-tuning produce structurally different weight matrices — LoRA introduces "intruder dimensions" linked to catastrophic forgetting. So full fine-tuning is still preferred when resources allow.

The big 2025 shift is RL with verifiable rewards (RLVR), especially for math/code/proofs where a checker provides binary correctness signals:

  • GRPO (DeepSeek-R1) — samples multiple responses per prompt, normalizes rewards within the group. No critic model needed. DeepSeek-R1-Zero went from 15.6% to 71.0% on AIME 2024 using purely rule-based rewards.
  • DAPO (ByteDance, NeurIPS 2025) — stabilizes GRPO for long reasoning chains. Trained Qwen2.5-32B to 50 AIME points with 50% fewer steps than DeepSeek-R1-Zero.
  • DPO remains the most cost-effective alignment method for most production cases (~2K preference pairs is enough). Variants: SimPO, KTO (binary feedback instead of pairs), ORPO (merges SFT + preference in one objective).

For theorem proving specifically, expert iteration dominates:

  1. Model generates proof attempts
  2. Lean verifier checks them
  3. Fine-tune on verified-correct proofs
  4. Repeat

Notable systems:

  • DeepSeek-Prover-V2 — expert iteration + GRPO + subgoal decomposition (88.9% MiniF2F)
  • Goedel-Prover-V2 — formalizes 1.64M NL math statements into Lean, iterative expert iteration with scaffolded difficulty (88.1% MiniF2F at pass@32)
  • STP — dual-agent self-play: a conjecturer generates barely-provable statements, a prover attempts them. Doubled previous best on LeanWorkbook.

What is taking place intuitively and mathematically in PEFT?

The core observation

A pretrained model has a weight matrix $W_0 \in \mathbb{R}^{d \times k}$ at each layer. Full fine-tuning learns an update $\Delta W$ of the same shape, so the new weight is $W = W_0 + \Delta W$. The matrix $\Delta W$ has $d \times k$ free parameters — millions or billions across all layers.

The empirical finding (Aghajanyan et al. 2020, "Intrinsic Dimensionality Explains the Effectiveness of Language Model Fine-Tuning") is that task-specific adaptation lives in a much lower-dimensional subspace than the full parameter space. You can project the update into a random low-dimensional subspace and still recover most of the fine-tuning performance. The "intrinsic dimension" of fine-tuning is surprisingly small — often a few thousand, even for models with hundreds of millions of parameters.

LoRA: the mechanism

Given that $\Delta W$ is effectively low-rank, LoRA parametrizes it directly as a low-rank factorization:

$$\Delta W = BA$$

where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times k}$, with $r \ll \min(d, k)$. Typical $r$ values are 4, 8, 16, 32.

During a forward pass, the output of the adapted layer is:

$$h = W_0 x + BAx$$

$A$ is initialized with small random values (usually Gaussian), $B$ is initialized to zero, so $\Delta W = 0$ at the start of training — the model begins exactly at the pretrained weights. Only $A$ and $B$ are trained; $W_0$ is frozen.

The parameter count drops from $d \times k$ to $r \times (d + k)$. For a typical transformer attention projection with $d = k = 4096$ and $r = 16$, that's $16.8\text{M} \to 131\text{K}$ — a 128x reduction per matrix.

Intuitively: you're saying "the pretrained model already knows almost everything needed; adaptation only needs to nudge the representations along a few directions." The rank $r$ controls how many independent directions of adjustment you allow.

Where LoRA is applied

Typically to the query and value projection matrices ($W_Q$, $W_V$) in attention layers. Some practitioners also include $W_K$, $W_O$, and the MLP layers. More target matrices = more expressiveness but more parameters. The original LoRA paper found $W_Q$ and $W_V$ sufficient for most tasks.

DoRA: decomposing magnitude and direction

A weight matrix $W$ can be decomposed as:

$$W = m \frac{V}{\|V\|_c}$$

where $m$ is a magnitude scalar (per column or per output neuron) and $V / \|V\|_c$ is the directional component (column-normalized).

DoRA's observation: full fine-tuning tends to make large directional changes with small magnitude changes. Standard LoRA couples these — adding $BA$ changes both magnitude and direction simultaneously, and it tends to produce correlated magnitude-direction updates that don't match what full fine-tuning does.

DoRA decouples them:

  • Learn magnitude $m$ as a free vector (one scalar per output neuron)
  • Apply LoRA only to the directional component $V$
$$W' = m' \frac{V + BA}{\|V + BA\|_c}$$

This gives the optimizer independent control over "how much" vs. "which way," matching the structure of full fine-tuning updates more faithfully. The result is better performance at the same rank, and the improvement is largest at low ranks where LoRA is most constrained.

QLoRA: a memory optimization, not a mathematical one

QLoRA is orthogonal to the above. It quantizes the frozen $W_0$ to 4-bit (using NormalFloat4 quantization), stores it compressed, dequantizes on the fly during the forward pass, and keeps the LoRA adapters $A$, $B$ in full precision (bfloat16). The backward pass computes gradients only for $A$ and $B$.

There is a precision loss from the 4-bit base, but it's small because the base weights are frozen — you never accumulate gradient errors in them. The adapters absorb the task-specific signal at full precision.

The "intruder dimensions" problem

The NeurIPS 2025 paper found something structural. Take the SVD of the adapted weight $W_0 + BA$. LoRA introduces new high-ranking singular vectors — directions that weren't prominent in $W_0$ and don't appear in full fine-tuning either. These "intruder dimensions" are artifacts of the low-rank constraint forcing the update into a subspace that doesn't align with the model's existing representational structure. They correlate with catastrophic forgetting: the model performs well on the new task but loses capability on the original distribution.

Full fine-tuning, unconstrained, finds updates that stay within the model's natural singular vector basis. LoRA, constrained to rank $r$, sometimes can't express the needed update within that basis and instead introduces orthogonal directions that interfere with existing representations.

This is the fundamental tradeoff: low rank saves memory and compute, but if the true $\Delta W$ has important components outside the rank-$r$ subspace, LoRA will either miss them or distort the representation to compensate.

The entire family rests on one bet: that the gap between "what the pretrained model knows" and "what the task needs" is low-dimensional. For most practical tasks, this bet pays off.

What sorts of data are shown to the training in LoRA, DoRA and so on? And how much? And if the answer is, you have it generate a Lean proof and then tell it "nope" or "yep", how is that backpropagated?

What the training data looks like

For SFT-based LoRA/DoRA, the data is just input-output pairs, formatted as the model would see them at inference. For theorem proving, a single training example might be:

Input:  "Prove that for all n : ℕ, 0 + n = n"
Output: "theorem zero_add (n : ℕ) : 0 + n = n := by
          induction n with
          | zero => rfl
          | succ n ih => simp [Nat.add_succ, ih]"

The loss is standard next-token cross-entropy on the output tokens, with the input tokens masked (no gradient signal from them). LoRA vs. full fine-tuning differs only in which parameters receive the gradient updates — the loss function and data format are identical.

For preference-based methods (DPO), the data is triples: a prompt, a preferred response, and a dispreferred response. For theorem proving you'd generate two proof attempts, check which one Lean accepts, and label accordingly.

Typical volumes for LoRA SFT: 1K–10K high-quality examples for domain adaptation. You can get results with a few hundred. More data helps but with sharply diminishing returns past ~5K for most tasks.

The harder question: how does "yep/nope" become gradients?

This is where GRPO (and REINFORCE-family methods generally) come in, and it's worth being precise about the mechanism because it's not obvious.

The setup

You have a language model $\pi_\theta$ that generates tokens autoregressively. Given a prompt $x$ (a theorem statement), it samples a response $y = (y_1, y_2, \ldots, y_T)$ (a proof attempt). The Lean checker returns a binary reward:

$$R(x, y) = \begin{cases} 1 & \text{proof checks} \\ 0 & \text{proof fails} \end{cases}$$

The question: the reward is a function of the entire completed sequence. It's not differentiable — Lean's type checker is an opaque binary oracle. How do you get gradients for $\theta$?

The policy gradient trick

The objective is to maximize expected reward:

$$J(\theta) = \mathbb{E}_{y \sim \pi_\theta(\cdot|x)}[R(x, y)]$$

You can't differentiate through the sampling process directly. But you can use the log-derivative trick (REINFORCE, Williams 1992):

$$\nabla_\theta J(\theta) = \mathbb{E}_{y \sim \pi_\theta(\cdot|x)}\left[R(x, y) \cdot \nabla_\theta \log \pi_\theta(y|x)\right]$$

The term $\log \pi_\theta(y|x)$ is just the sum of log-probabilities of the tokens the model actually generated:

$$\log \pi_\theta(y|x) = \sum_{t=1}^{T} \log \pi_\theta(y_t | x, y_{<t})$$

This is differentiable with respect to $\theta$ — it's the same computation as the standard language modeling forward pass. You already compute these logits during generation.

So the gradient is:

$$\nabla_\theta J \approx R(x, y) \cdot \sum_{t=1}^{T} \nabla_\theta \log \pi_\theta(y_t | x, y_{<t})$$

Intuitively: if the proof checked ($R=1$), increase the probability of every token in that sequence. If it failed ($R=0$), the gradient is zero — that attempt contributes nothing. The model learns by reinforcing successful generation paths.

The problem with raw REINFORCE

This has terrible variance. Most proofs fail, so most samples contribute zero gradient. The successful ones push the model toward those specific token sequences, but the signal is noisy — a proof might have succeeded for reasons unrelated to most of its tokens.

GRPO's solution: relative rewards within a group

For each prompt $x$, sample $G$ responses $\{y_1, \ldots, y_G\}$ (typically $G = 8$ to $64$). Compute rewards $\{R_1, \ldots, R_G\}$. Normalize within the group:

$$\hat{A}_i = \frac{R_i - \text{mean}(\{R_j\})}{\text{std}(\{R_j\})}$$

The gradient becomes:

$$\nabla_\theta J \approx \frac{1}{G} \sum_{i=1}^{G} \hat{A}_i \cdot \nabla_\theta \log \pi_\theta(y_i | x)$$

Now a proof that succeeds when most others fail gets a large positive advantage. A proof that fails when all others also fail gets advantage near zero. A proof that fails when others succeed gets a negative advantage — actively pushing the model away from that token sequence.

This is why GRPO works with binary rewards: the relative signal within the group converts a sparse binary signal into a richer gradient. If you sample 16 proofs and 2 check, those 2 get boosted and the other 14 get suppressed, proportionally.

There's also a KL penalty term against a reference policy $\pi_{\text{ref}}$ to prevent the model from collapsing to a narrow set of proof strategies:

$$\mathcal{L} = -\hat{A}_i \cdot \frac{\pi_\theta(y_i|x)}{\pi_{\text{ref}}(y_i|x)} + \beta \cdot D_{\text{KL}}(\pi_\theta \| \pi_{\text{ref}})$$

The full loop for theorem proving

  1. Sample: Given theorem statement $x$, generate 16 proof attempts from $\pi_\theta$
  2. Verify: Send each to Lean's type checker. Get binary rewards.
  3. Normalize: Compute group-relative advantages $\hat{A}_i$
  4. Update: Run the forward pass on all 16 sequences, compute log-probs, weight by advantages, backpropagate through the LoRA/DoRA adapters (or full weights)
  5. Repeat with new theorem statements

The Lean checker is never differentiated through. It's purely an oracle. All the differentiable computation happens inside the model's own forward pass on sequences it already generated. The reward just tells you which direction to push.

Why this works despite being so crude

The model already knows a lot about Lean syntax and proof strategies from pretraining. The RL signal doesn't need to teach it Lean from scratch — it just needs to shift probability mass from almost-correct proofs to actually-correct ones. The intrinsic dimensionality of that adjustment is low, which is exactly why LoRA works here too.

How can you perturb the model to better understand a certain genre of query? i.e. what if the way people ask math questions is a little different than standard chats? Can you provide a large corpus of those maybe? Without proofs, just stylistic language?

This is the distinction between adapting the model's input comprehension (understanding math-register language) versus its output capability (generating proofs). Different methods apply.

Continued pretraining

The most direct approach. You take the frozen pretrained model and do additional unsupervised language modeling (next-token prediction) on a large corpus of mathematical text. No labels needed — just raw text in the target register.

This shifts the model's token distribution, attention patterns, and internal representations toward mathematical discourse. After continued pretraining, the model will:

  • Parse "WLOG assume $n > 0$" without stumbling
  • Understand that "clearly" means "I'm not going to prove this"
  • Handle mixed notation/prose fluently
  • Recognize implicit structure ("by the above," "the claim follows")

Typical data volumes: 1B–30B tokens for standard continued pretraining. But the ICLR 2025 synthetic continued pretraining paper showed ~1M tokens can work if the data is sufficiently targeted and augmented with entity-relationship paraphrases.

The catastrophic forgetting problem: if you only train on math text, the model loses general capability. The standard fix is mixing: 50–80% domain data, 20–50% replayed general data (or data from the original pretraining distribution).

What corpora exist

Large-scale mathematical text:

  • OpenWebMath (~15B tokens) — math web pages filtered from Common Crawl, cleaned of boilerplate. Probably the most used open corpus for math continued pretraining.
  • RedPajama and Proof-Pile (EleutherAI) — includes arXiv, Math StackExchange, textbooks. Proof-Pile II is ~55B tokens.
  • arXiv bulk access — all papers, ~2M+ math papers. Raw LaTeX source. Very heavy on formal notation mixed with terse prose.
  • Math StackExchange / MathOverflow data dumps — Q&A format, informal-to-semiformal register, exactly the genre you're describing. ~3M questions on MSE.
  • ProofWiki, Mathlib docs, Lean Zulip archives — for the formal verification register specifically.
  • AlgebraicStack (used by Llemma) — 11B tokens, curated blend of arXiv, StackExchange, open textbooks, and formal proof libraries.

The Llemma model (EleutherAI, 2023) is the canonical example of this approach: they took Code Llama 7B/34B and did continued pretraining on AlgebraicStack (11B tokens). No fine-tuning, no RL — just more pretraining on math text. It substantially improved performance on math benchmarks purely from better input comprehension and domain familiarity.

The register issue specifically

Mathematical discourse has several distinct registers, and they differ more than people realize:

RegisterExampleWhere to find it
Textbook formal"Let $G$ be a finite group. Then the order of every element divides $|G|$."Open textbooks, Proof-Pile
Research paper"The result follows from a straightforward application of Thm. 3.2 combined with the spectral bound established in §4."arXiv
Student question"I don't get why you can just swap the limit and the integral here??"Math StackExchange
Lean/formaltheorem dvd_of_mem_order (h : a ∈ G) : orderOf a ∣ card GMathlib, Lean Zulip
Informal expert"This is basically just Yoneda in disguise, right?"MathOverflow, Zulip, Twitter/Mastodon

If your target users write in the "student question" register but your model was pretrained mostly on arXiv, there's a distribution mismatch on the input side. Continued pretraining on Math StackExchange data specifically would close that gap without needing any labeled proof data.

How little data can work

You don't always need billions of tokens. Two lighter approaches:

1. Synthetic continued pretraining (~1M tokens). Take a small set of domain documents, use a larger model to generate paraphrases that express the same concepts in varied phrasings:

  • "Prove that the sum of two even numbers is even"
  • "Show that adding two even integers gives an even result"
  • "Why is it true that even + even = even?"
  • "Let $m = 2a$ and $n = 2b$. Verify that $m + n$ is even."

This creates an entity-relationship graph across phrasings that helps the model generalize from a small seed corpus.

2. Prefix/prompt tuning. Instead of modifying any model weights, learn a small set of continuous "virtual tokens" (soft prompts) prepended to the input. These are trained on math text to steer the model's attention patterns toward mathematical parsing. Parameter count: ~20K (tens of virtual tokens × embedding dimension). This is even cheaper than LoRA but less expressive.

Combining input and output adaptation

The full pipeline for something like Leanstral is likely:

  1. Continued pretraining on math + Lean corpus (input comprehension)
  2. SFT on verified proof pairs (output capability)
  3. GRPO/RL with Lean checker rewards (output refinement beyond training data)

Step 1 is what you're asking about — and it's often underappreciated. A model that can't parse the question register will struggle regardless of how much RL you throw at the output side.

These all require access to the full model weights then?

Yes. Continued pretraining, SFT, LoRA, DoRA, GRPO — all of them require the weights because you're either modifying them directly or computing gradients through the forward pass.

The approaches that don't require weights:

Prompt-based methods (API-only)

Few-shot prompting. Prepend examples of the target register to every query. No training, no weights — you're just using context to shift the model's output distribution. Costs tokens at inference time but works surprisingly well for register adaptation.

Retrieval-Augmented Generation (RAG). At query time, retrieve relevant math text from a corpus and inject it into the context. The model never changes — you're giving it reference material on the fly. For register adaptation specifically, you'd retrieve examples of similar questions and their answers.

System prompts / meta-prompting. "You are a mathematician. Users will ask questions using informal mathematical language. Parse their intent carefully." Crude but non-zero.

Distillation (API access to a teacher, weights of a student)

If you have API access to a strong model (e.g. Claude, GPT) but weights of a smaller open model (e.g. Llama, Mistral), you can:

  1. Send math questions to the large model via API
  2. Collect its responses
  3. Fine-tune the small model on those pairs using LoRA/SFT

You never touch the large model's weights. You do need weights for the small model you're training.

The hard constraint

There's no way around it: if you want the model itself to change — to internalize a new register, shift its probability distribution, update its representations — somebody needs the weights. The gradient has to flow through the model's parameters.

The API-only methods don't change the model. They change what the model sees at inference time. That's a meaningful distinction — you're compensating for the model's gaps with context engineering rather than fixing them. It works, but it has limits: context windows are finite, retrieval adds latency, and the model's underlying biases remain.

This is why open-weight models (Llama, Mistral, Qwen) matter so much for specialized domains. Without weights, you're limited to steering; with weights, you can actually teach.

Can any of these adaptation methods run on an M3 Max laptop in reasonable time, like a few days?

Yes, within limits.

What fits

An M3 Max has 48GB or 96GB unified memory (CPU and GPU share it). The Metal backend in MLX (Apple's ML framework) and llama.cpp can use all of it. This is the main advantage over discrete GPUs — a 4090 has 24GB, and you'd need multi-GPU setups for anything the M3 Max handles in one pool.

QLoRA on a 7–8B model: This is the sweet spot. Quantize the base model to 4-bit (~4GB), keep LoRA adapters in bfloat16 (~50–100MB for rank 16). Total memory: ~6–8GB. With MLX or Hugging Face's mlx-lm fine-tuning, you can process ~1K–5K examples in hours, not days.

QLoRA on a 13–14B model: ~8–10GB quantized. Still fine on 48GB. A few hours to a day for typical SFT datasets.

QLoRA on a 32–34B model: ~18–20GB quantized. Fits on 48GB, comfortable on 96GB. Training is slower — expect 1–3 days for a few thousand examples.

QLoRA on 70B: ~35–40GB quantized. Only fits on the 96GB configuration, and tightly. Training is feasible but slow — several days. You'd want to be sure of your data before committing.

Full fine-tuning of 7B in bfloat16: The model alone is ~14GB, but optimizer states (Adam) roughly 3x that — so ~50–60GB. Possible on 96GB, too tight on 48GB. Slower than QLoRA but avoids the intruder dimension problem.

What doesn't fit

Full fine-tuning of anything above ~7B. GRPO/DAPO — these require holding multiple copies of the model (policy + reference) and generating dozens of rollouts per prompt. Even with a 7B model, the throughput is too low to finish the millions of rollouts needed in any reasonable time.

Continued pretraining on billions of tokens — even with QLoRA, the sheer data volume makes this a multi-week affair on laptop hardware.

Tooling

MLX is the path of least resistance on Apple Silicon. mlx-lm has built-in LoRA/QLoRA fine-tuning with a single command:

  --model mistralai/Mistral-7B-v0.3 \
  --train \
  --data ./math-qa-data \
  --batch-size 4 \

Unsloth also supports Apple Silicon and claims 2x speed improvements over stock implementations through fused kernels, though Metal support is less mature than their CUDA path.

So: QLoRA + 7–8B + MLX is the comfortable local option. You can genuinely adapt a model to a math register on your laptop in an afternoon.

Part 2: Math register fine-tuning

Model choice: Qwen2.5-7B-Instruct has the strongest math baseline among 7B-class models. At 4-bit quantization it's ~4GB, leaving plenty of room in 36GB for training.

Data: MetaMathQA (~395K math Q&A pairs bootstrapped from GSM8K and MATH, in the informal question-asking register).

# Download and prepare the data
python - <<'PYEOF'
from datasets import load_dataset
import json, os, random

# MetaMathQA: math questions in natural language with step-by-step answers
ds = load_dataset("meta-math/MetaMathQA", split="train")

# Shuffle and take a manageable subset (5K trains fast, ~2hrs)
indices = list(range(len(ds)))
random.seed(42)
random.shuffle(indices)

os.makedirs("math-data", exist_ok=True)

def to_chat(example):
    return {"messages": [
        {"role": "user", "content": example["query"]},
        {"role": "assistant", "content": example["response"]}
    ]}

# 5000 train, 500 validation, 500 test
splits = [("train.jsonl", indices[:5000]),
          ("valid.jsonl", indices[5000:5500]),
          ("test.jsonl",  indices[5500:6000])]

for fname, idxs in splits:
    with open(f"math-data/{fname}", "w") as f:
        for i in idxs:
            f.write(json.dumps(to_chat(ds[i])) + "\n")

print(f"Wrote {len(splits)} splits to math-data/")

Part 3: Lean theorem proving fine-tuning

This is a separate adapter trained on Lean 4 tactic proofs.

Data: LeanWorkbook (from InternLM) contains Lean 4 theorem statements with proofs.

python - <<'PYEOF'
from datasets import load_dataset
import json, os, random

# LeanWorkbook: Lean 4 theorem-proof pairs
ds = load_dataset("InternLM/Lean-Workbook", split="train")

os.makedirs("lean-data", exist_ok=True)

examples = []
for row in ds:
    # Filter for entries that have complete proofs
    stmt = row.get("formal_statement", "")
    proof = row.get("proof", "")
    if not stmt or not proof:
        continue
    examples.append({"messages": [
        {"role": "user", "content": f"Prove the following in Lean 4:\n\n{stmt}"},
        {"role": "assistant", "content": proof}
    ]})

random.seed(42)
random.shuffle(examples)

splits = [("train.jsonl", examples[:5000]),
          ("valid.jsonl", examples[5000:5500]),
          ("test.jsonl",  examples[5500:6000])]

for fname, data in splits:
    with open(f"lean-data/{fname}", "w") as f:
        for ex in data:
            f.write(json.dumps(ex) + "\n")

print(f"Wrote {len(examples)} total examples, split to lean-data/")

import json, subprocess, tempfile, os
from mlx_lm import load, generate

def check_lean_proof(statement, proof):
    """Send a proof to Lean 4 and check if it type-checks."""
    lean_code = f"""import Mathlib\n\n{statement} := by\n{proof}\n"""

    with tempfile.NamedTemporaryFile(
        mode='w', suffix='.lean', delete=False, dir='.'
    ) as f:
        f.write(lean_code)
        fname = f.name

    try:
        result = subprocess.run(
            ["lean", fname],
            capture_output=True, text=True, timeout=60
        )
        return result.returncode == 0
    except subprocess.TimeoutExpired:
        return False
    finally:
        os.unlink(fname)

def eval_lean_model(model_path, adapter_path=None, label="model"):
    model, tokenizer = load(model_path, adapter_path=adapter_path)

    with open("lean-data/test.jsonl") as f:
        tests = [json.loads(line) for line in f]

    proved = 0
    attempted = min(50, len(tests))  # Lean checking is slow

    for i, test in enumerate(tests[:attempted]):
        statement = test["messages"][0]["content"].replace(
            "Prove the following in Lean 4:\n\n", ""
        )
        prompt = tokenizer.apply_chat_template(
            [{"role": "user",
              "content": test["messages"][0]["content"]}],
            tokenize=False, add_generation_prompt=True
        )
        response = generate(
            model, tokenizer, prompt=prompt, max_tokens=1024
        )

        if check_lean_proof(statement, response):
            proved += 1
            print(f"  [{label}] {i+1}: PROVED")
        else:
            print(f"  [{label}] {i+1}: failed")

    print(f"\n{label}: {proved}/{attempted} "
          f"proved ({100*proved/attempted:.1f}%)")
    return proved

print("=== Base model ===")
base = eval_lean_model(
    "Qwen/Qwen2.5-7B-Instruct", label="base"
)

print("\n=== Lean-tuned model ===")
tuned = eval_lean_model(
    "Qwen/Qwen2.5-7B-Instruct",
    adapter_path="./adapters-lean", label="lean-tuned"
)

print(f"\nDelta: {tuned - base} more proofs from fine-tuning")

Caveats

  • The Lean evaluation script is naive. It assumes Mathlib is available in the Lean environment, generates one attempt per problem (pass@1), and uses a 60-second timeout. Real systems generate 16–64 attempts per theorem. You'll get low absolute numbers but the relative improvement between base and tuned is what matters.
  • The LeanWorkbook dataset structure may differ from what's assumed here — field names might be natural_language and lean_proof instead of formal_statement and proof. Check the dataset card and adjust the field names in the data prep script.
  • Watch validation loss during training. If it starts climbing while training loss keeps dropping, you're overfitting. Stop early and use the checkpoint with lowest validation loss.
